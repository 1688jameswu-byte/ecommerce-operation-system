# AGENTS.md

## 1. Codex 工作规则

本项目是公司内部电商运营中心，涉及 TEMU 运营数据、1688业务、薪资绩效、运营闭环、权限管理和运营工具。

Codex 在执行任何任务前，必须先判断任务类型，再决定需要读取哪些文档和代码。

不要无脑读取全部文档，避免浪费上下文、降低速度和抓错重点。

## 3. 修改边界

开发时必须遵守以下边界：

* 不要重构无关功能。
* 不要删除已有业务数据。
* 不要破坏现有路由。
* 不要破坏现有权限逻辑。
* 不要影响已上线页面。
* 新字段必须兼容旧数据。
* 不要因为一个新模块改坏旧模块。
* 不要把临时测试逻辑写进正式代码。
* 不要把真实账号密码、数据库密码、API Key 写进代码。
* 不要提交真实 `.env` 文件。

---

## 4. 数据存储边界

当前数据策略：

```text
旧模块：继续使用 JSON
1688新业务：直接使用 PostgreSQL
```

旧模块包括：

* TEMU运营数据
* 薪资绩效数据
* 运营闭环数据
* 基础资料数据

1688业务可以使用 PostgreSQL，但不要因此迁移旧模块。

重要限制：

* 不要一次性把全系统迁移到 PostgreSQL。
* 不要影响旧的 JSON 持久化逻辑。
* 前端不能直接连接 PostgreSQL。
* PostgreSQL 必须通过后端 API 访问。
* 数据库连接信息必须来自环境变量。

---

## 5. 1688业务边界

1688业务当前定位：

```text
半自动上架资料库 + 人工上架任务闭环
```

当前不做：

* 1688 API 全自动上架
* 浏览器模拟自动发布
* 爬虫式自动发布
* 自动发布/API上架菜单
* 客户管理
* 询盘管理
* 订单管理
* 数据分析

当前优先做：

* 产品库
* 上架任务
* 图片素材
* 供应商管理
* 1688设置

SKU 不做独立菜单，放在产品详情页里。

样品库不单独做，因为当前拿回来的样品基本都是为了上架。

---

## 6. 权限边界

系统需要支持：

* 不同账号看到不同菜单
* 不同账号看到不同店铺
* 不同账号看到不同产品和任务
* 无权限的按钮、卡片、快捷入口不显示
* 后端也必须校验权限，不能只靠前端隐藏

敏感信息包括：

* 供应商成本
* 拿货价
* 数据库连接信息
* API Key
* SESSION_SECRET
* 员工隐私数据

普通业务员不应看到供应商核心成本和拿货价。

---

## 7. 页面体验要求

后台管理页面应适合长期办公：

* 优先浅色主题
* 表格、表单、弹窗清晰
* 大数据页面不能一次性加载全部数据
* 需要分页、筛选、懒加载或虚拟滚动
* 编辑操作优先弹窗或抽屉
* 删除、停用、覆盖等危险操作必须确认
* 删除按钮不要离上传按钮太近

运营大屏可以继续保留深色风格。

---

## 8. 开发流程要求

每次修改前：

1. 判断任务类型。
2. 按文档读取规则读取必要文档。
3. 查找相关代码。
4. 最小范围修改。
5. 兼容旧数据。
6. 避免影响无关模块。
7. 完成后运行构建检查。

常规检查命令：

```bash
npm run build
```

如果项目支持 TypeScript 单独检查，也需要确保 TypeScript 通过。
